# Technopath — правила работы с проектом

Этот файл обязателен для всех задач Codex в данном репозитории. Перед изменениями также прочитать `Docs/Architecture.md`.

## Основные принципы

- Сначала изучить существующие файлы, активную сцену, префабы, ScriptableObject и Console. Не создавать параллельную систему, пока не проверено, что подходящей системы ещё нет.
- Сохранять проект чистым: одна ответственность на систему, явные зависимости, понятные имена и минимальная связанность.
- Не вводить фреймворк, пакет, глобальный сервис или новый архитектурный слой без реальной необходимости.
- Сохранять пользовательские изменения. Не удалять и не перезаписывать незнакомые ассеты и настройки без явного согласования.
- Если требование неоднозначно и выбор заметно повлияет на архитектуру или игровой дизайн, сначала обсудить решение.

## Unity Editor и MCP

- Для чтения и изменения сцен, GameObject, компонентов, префабов, ScriptableObject-ассетов, материалов, UI и настроек Unity использовать Unity MCP.
- Перед изменением Editor-состояния прочитать связанные MCP-ресурсы: состояние редактора, проект, активную сцену и при необходимости Console.
- Не писать вручную Unity YAML для `.unity`, `.prefab`, `.asset`, `.mat` и других сериализованных Unity-ассетов.
- Не заменять доступную MCP-операцию одноразовым Editor-скриптом. Editor tooling допустим только как полезный постоянный инструмент проекта.
- Не собирать сцену или UI целиком из кода во время запуска, если это авторский контент, который должен быть видим и редактируем в Unity Editor.
- После изменений C# дождаться завершения компиляции и проверить Unity Console.
- После изменения сцены или префаба сохранить его и повторно прочитать результат через MCP.
- Перед использованием незнакомого или версионно-зависимого Unity API проверить его через `unity_reflect`; при необходимости открыть официальную документацию через `unity_docs`.
- Не запускать Play Mode, тесты или сборку, если редактор компилирует или обновляет ассеты.

## Разделение данных и поведения

- `ScriptableObject` содержит определения контента и конфигурацию, но не изменяемое состояние игровой сессии.
- Prefab содержит состав объекта, компоненты и визуальное представление.
- Обычный C# runtime-объект или компонент хранит состояние конкретного экземпляра во время игры.
- Save-модель хранит только сериализуемые данные прогресса и не зависит напрямую от объектов сцены.
- MonoBehaviour отвечает за Unity lifecycle и представление; доменная логика по возможности остаётся в обычных C# классах.
- Конфигурационные ScriptableObject во время игры рассматриваются как read-only.

## Зависимости и коммуникация

- Зависимости передавать явно через сериализованные ссылки, фабрики, конструкторы обычных C# классов или согласованный composition root.
- Не использовать `FindObjectOfType`, `GameObject.Find`, строки-пути и глобальные mutable singleton как основной способ связывания систем.
- Не добавлять service locator или DI-контейнер до отдельного архитектурного решения.
- События применять для действительно ослабленной связи. Не превращать глобальную шину событий в скрытый control flow.
- Не смешивать игровую логику, UI, сохранения, ввод и загрузку сцен в одном классе.

## Сцены, префабы и UI

- Переиспользуемый игровой или UI-объект оформлять как prefab.
- Сцена описывает композицию уровня или этап приложения, а не служит хранилищем глобальных сервисов без управляемого lifecycle.
- Prefab variants использовать для визуальных и конфигурационных вариаций общего объекта; различное поведение оформлять композиционно.
- UI получает подготовленное состояние через presenter/controller/view-model слой и не управляет доменной логикой напрямую.
- Навигация UI должна быть централизованной и предсказуемой; окна не должны самостоятельно искать и открывать друг друга.
- Настраиваемые ссылки должны быть видимы в Inspector и валидироваться как можно раньше.

## Структура и качество кода

- Runtime-код не должен зависеть от пространства имён `UnityEditor`.
- Editor-код хранить только в папках `Editor` или Editor-only assembly definition.
- Избегать больших manager-классов. При росте ответственности разделять систему по поведению, а не только по размеру файла.
- Публичный API делать минимальным; сериализованные поля по умолчанию оставлять `private`.
- Не создавать абстракцию ради одного вызова. Добавлять интерфейс там, где есть реальная граница, несколько реализаций или необходимость изолированного тестирования.
- Для важных правил доменной логики добавлять Edit Mode тесты; для интеграции компонентов и сцен — Play Mode тесты по необходимости.
- Новые папки и имена должны следовать соглашениям из `Docs/Architecture.md`.

## Обязательная проверка перед завершением задачи

1. Unity завершил импорт и компиляцию.
2. В Console нет новых ошибок.
3. Изменённые сцены и префабы сохранены.
4. Ссылки Inspector назначены; отсутствующие обязательные ссылки выявляются понятной ошибкой или валидацией.
5. Конфигурация отделена от runtime-состояния.
6. Не появились ручные YAML-ассеты, временные Editor-скрипты или ненужные глобальные зависимости.
7. Архитектурное решение отражено в `Docs/Architecture.md` или отдельном ADR, если оно влияет на дальнейшую разработку.
