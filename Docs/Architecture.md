# Technopath — архитектура проекта

Статус: живой документ. Обновлять после согласования решений, влияющих на структуру проекта.

Утверждённые правила игры и боевой системы находятся в `Docs/GameDesign.md`. Архитектурная реализация не должна менять их смысл без предварительного обсуждения.

Порядок реализации и критерии готовности итераций находятся в `Docs/DevelopmentRoadmap.md`.

## Текущий контекст

- Движок: Unity `6000.5.1f1`.
- Render Pipeline: Universal Render Pipeline.
- Жанр: пошаговый roguelike с управлением формацией роботов.
- Исходный шаблон: 2D; окончательный визуальный способ представления ещё не зафиксирован.
- Основная платформа редактора и текущей сборки: Windows / `StandaloneWindows64`.
- Интеграция редактора: MCP for Unity.
- Текущая тестовая сцена: `Assets/_Project/Scenes/CombatSandbox.unity`; она используется для пошаговой проверки боевого ядра и не является финальной игровой сценой.

## Архитектурная цель

Проект должен позволять добавлять и настраивать контент через ScriptableObject, prefab и Inspector без изменения системного кода. Игровая логика, данные конфигурации, runtime-состояние и визуальное представление разделяются.

```text
ScriptableObject Definition/Config
              |
              v
Runtime Model / Game Logic <---- Input and application commands
              |
              v
Unity Components / Prefab ----> UI presentation
              |
              v
           Save Data
```

### Боевое поле

- `Combat.Board` — чистая runtime-модель двух независимых сеток `3×3`; она не зависит от сцены, prefab и ввода.
- Координата хранит строку и столбец, а индекс используется только как компактное отображение в массив из девяти клеток.
- Клетка различает пустое состояние, юнита и временный объект. Конкретный экземпляр хранится по runtime ID, а не ссылкой на GameObject.
- `Combat.Presentation` связывает модель с prefab клеток и фигур. Ссылки на девять клеток каждой стороны назначаются явно через Inspector.
- Стартовая расстановка создаётся фабрикой по seed: Технопат занимает центр, три робота — уникальные случайные клетки своей половины.
- Pointer input преобразует ввод мыши в выбор клетки, но не содержит правил поля. Горячие клавиши передают те же команды через отдельный слой ввода; touch и геймпад пока не реализуются.
- `Combat.Rules.PlayerTurnModel` атомарно применяет перемещение или swap, списывает общее ОД и формирует упорядоченный результат автоатак. Presentation должна только проигрывать уже рассчитанный результат.
- Самостоятельная активация отмечается только у инициатора. Юнит, перемещённый вследствие swap, сохраняет право позднее инициировать собственное действие.
- `Combat.Round` управляет фазами и хранит полные намерения мутантов, включая скрытую клетку. Presentation получает из намерения только видимую атаку и точный урон.
- Вражеская фаза разрешается упорядоченной цепочкой: фиксированный приоритет типа, случайный tie-break внутри приоритета, атака, затем заранее выбранный шаг. Исход боя определяется только после завершения цепочки.
- Броня является отдельным запасом экземпляра юнита: сначала поглощает входящий урон, затем остаток переносится в здоровье. В начале фазы игрока броня живых союзников восстанавливается до максимума; броня мутантов не восстанавливается автоматически.
- Базовые боевые изменения записываются в ограниченную FIFO-очередь событий. Лимит цепочки защищает от бесконечных реакций, а presentation читает готовую последовательность для подробного debug-журнала.
- Архетип робота описывается `RobotArchetypeDefinition`: базовые характеристики, видимое правило условной способности, момент события, область реакции и ограничение частоты. Runtime-трекер отделён от ScriptableObject и хранит использования за действие, фазу и бой.
- Зарядовые статусы хранятся в runtime-коллекции экземпляра, а их неизменяемые параметры описываются `StatusDefinition`. Один тик рассчитывает эффект из всех текущих зарядов и затем снимает один заряд.
- `RobotModuleDefinition` описывает неизменяемый модуль: слот, редкость, уровень, совместимость, изменения характеристик и при необходимости условную способность. Во время боя definition остаётся read-only.
- `RobotLoadout` является обычным runtime-объектом и собирает ровно одно Ядро, один Процессор и три Модификатора; пустые слоты разрешены. Он рассчитывает итоговые характеристики и сохраняет список источников каждого изменения.
- Ядро заменяет основную способность архетипа, Процессор добавляет утилитарную условную способность. Эти правила вычисляются loadout-моделью, а не UI.
- `RobotLoadoutPresetDefinition` хранит редактируемый тестовый шаблон сборки. Presenter преобразует шаблоны в runtime-loadout и передаёт их боевой модели явно по ID экземпляра юнита.
- `LoadoutComparisonPanel` оформлен отдельным prefab и показывает две сборки одного архетипа без владения боевыми правилами.
- `RobotInspectionData` является подготовленной presenter-моделью для UI и не раскрывает панели внутренние боевые коллекции. `RobotInspectionPanel` оформлен отдельным prefab: наведение получает компактные данные, а выбор закрепляет расширенную карточку.
- `RunState` хранит изменяемое состояние текущего забега, включая детали и инвентарь модулей. `BattleRewardGenerator` атомарно формирует награду и добавляет её в это состояние; `VictoryRewardPanel` только отображает уже выданный результат.
- `CampRobotState` хранит пережившее бой здоровье робота и его runtime-loadout. Команды `RunState` выполняют установку, замену, снятие, разбор и ремонт атомарно; `CampPanel` реализует drag-and-drop и не меняет коллекции напрямую.
- `RestStop` является отдельной сценой этапа забега. Переход загружает её additively, явно передаёт `RunState` в `RestStopController`, после чего выгружает боевую сцену; глобальный mutable singleton и `DontDestroyOnLoad` для передачи состояния не используются.

Направление зависимостей уточняется для каждой системы, но доменная логика не должна зависеть от UI или конкретной сцены.

## Модель данных

### Definitions

Неизменяемые описания контента: персонажи, противники, способности, предметы, уровни и другие игровые сущности. Реализуются ScriptableObject-ассетами и могут содержать ссылки на визуальные ресурсы и prefab.

### Configs

Настройки систем и баланса: темп игры, сложность, экономика, звук, UI и прочие параметры. Config-ассеты считаются read-only во время выполнения.

### Runtime State

Изменяемое состояние конкретной сессии или экземпляра: здоровье, cooldown, инвентарь, текущая цель, прогресс уровня. Хранится в runtime-моделях и компонентах, но не в исходных ScriptableObject.

### Save Data

Версионируемые сериализуемые DTO без ссылок на GameObject, Component и сценовые экземпляры. Связь с definitions выполняется по устойчивым идентификаторам.

## Предварительная карта сцен

Окончательно определить после утверждения игрового цикла.

```text
Bootstrap  — composition root, запуск приложения и постоянные сервисы
MainMenu   — главное меню и настройки
Gameplay   — игровой контекст или оболочка загрузки уровней
```

Дополнительные сцены (`Loading`, `MetaGame`, отдельные уровни) добавляются только при подтверждённой необходимости. `DontDestroyOnLoad` не используется как неявное хранилище случайных manager-объектов.

## Prefab-стратегия

- Prefab является основной единицей переиспользуемой Unity-композиции.
- Корневой prefab содержит ясно определённый основной компонент и минимально необходимую публичную поверхность.
- Вложенные prefabs используются для независимых визуальных или функциональных модулей.
- Prefab variants применяются для вариаций общей основы, если различия не требуют отдельного поведения.
- Данные баланса не дублируются в каждом prefab, если они принадлежат definition/config asset.
- Авторский игровой объект или UI-элемент не считается prefab-композицией, если prefab содержит только один компонент, который целиком строит представление во время выполнения.
- Изменяемый состав отряда не размещается в сцене как постоянные конкретные роботы: сцена содержит явные платформы/слоты, а presenter создаёт в них назначенные prefab-представления роботов согласно runtime-состоянию.
- Цель presentation-слоя — позволить заменить placeholder-графику через prefab, Sprite/Model, Material, Animator и UI Style без изменения доменных правил.

Предварительные группы:

```text
Prefabs/
  Characters/
  Enemies/
  Gameplay/
  Environment/
  UI/
  Systems/
```

## UI

- Текущая технология продуктового интерфейса — uGUI. `OnGUI` сохраняется только для временных debug-панелей и не используется как основа игровых экранов.
- UI-представление не содержит правил игры.
- View отвечает за отображение и передачу пользовательских действий.
- Presenter/controller/view-model преобразует состояние игры в данные представления.
- Навигация между экранами управляется отдельным координатором.
- Переиспользуемые экраны и элементы оформляются prefabs.
- Статическая композиция экрана существует в сцене или экранном prefab и полностью видна в Unity Editor.
- Динамические списки и игровые сущности создаются из prefab в явные контейнеры; контейнеры, шаблоны и обязательные ссылки назначаются через Inspector.
- Экран собирается как `View + Presenter/Controller + item prefabs`: View отображает подготовленные данные и передаёт действия, но не меняет доменное состояние напрямую.
- Навигационный координатор управляет открытием экранов и модальных окон. Он не содержит боевых, экономических или контентных правил.
- Текущий целевой ввод — мышь и клавиатура с горячими клавишами. Hover допустим для дополнительных подсказок, поскольку touch и геймпад отложены до отдельной продуктовой итерации.

### Product-ready presentation standard

Экран считается готовым к последующей подстановке графики, только если:

1. Его статическая структура видна и редактируется в Unity Editor.
2. Повторяемые элементы представлены отдельными prefabs.
3. Runtime создаёт только изменяемые экземпляры из назначенных prefabs, а не строит авторскую вёрстку.
4. Все обязательные ссылки назначены через Inspector и валидируются до использования.
5. View не рассчитывает урон, экономику, награды, ремонт или другие игровые правила.
6. Замена placeholder-визуала не требует изменения доменной логики.
7. Экран проверен мышью и заявленными горячими клавишами на целевом PC-разрешении.

Исправление существующего presentation выполняется вертикальными срезами. Сначала согласуется структура одного экрана, затем создаются его сцена и prefabs, подключается существующая логика, выполняется Play Mode проверка и только после подтверждения решение переносится на следующий экран.

### Текущий боевой UI

- `BattleUIScreen.prefab` является редактируемой uGUI-композицией боевого HUD, журнала, компактной подсказки и подробной карточки робота.
- `BattleUIScreen` получает сценовый `BattlefieldPresenter` одной Inspector-ссылкой и передаёт его дочерним View; prefab не ищет боевую модель глобально.
- `CombatLogEntry.prefab` и `ModifierInspectionItem.prefab` являются динамическими повторяемыми элементами соответствующих контейнеров.
- `BattleHudView`, `CombatLogView` и `RobotInspectionPanel` только отображают подготовленное состояние и передают команды presenter.
- `BattleKeyboardInput` дублирует частые команды: `Space` завершает ход, `L` сворачивает журнал, `Escape` закрывает инспекцию.
- `DebugBattleControlsView` явно отделяет кнопку принудительной победы от продуктовых боевых правил.

## Composition Root и сервисы

Точная реализация ожидает согласования. Предпочтительное направление:

- одна явная точка сборки зависимостей в Bootstrap-контексте;
- интерфейсы на границах ввода, сохранений, аудио, загрузки и времени только там, где это даёт практическую пользу;
- отсутствие неуправляемых глобальных mutable singleton;
- явно определённый lifecycle каждого сервиса.

DI-контейнер пока не выбран. До отдельного решения не добавлять сторонний контейнер.

## События

- Локальное прямое взаимодействие предпочтительнее глобального события.
- C# events/signals используются для уведомлений между слабо связанными участниками.
- ScriptableObject event channels не вводятся автоматически: сначала требуется конкретный сценарий и анализ скрытых зависимостей.
- Подписки должны освобождаться симметрично и не переживать владельца.

## Предварительная структура Assets

Структуру вводить постепенно вместе с реальными системами, не создавая пустые каталоги заранее.

```text
Assets/
  _Project/
    Art/
    Audio/
    Content/
      Definitions/
      Configs/
    Prefabs/
    Scenes/
    Scripts/
      Runtime/
      Editor/
      Tests/
    UI/
  ThirdParty/
```

При миграции существующих ассетов учитывать GUID и перемещать их через Unity Editor/MCP.

## Идентификаторы контента

Definitions, участвующие в сохранениях или внешних таблицах, должны иметь устойчивый уникальный ID. Имя файла или отображаемое название не должны быть единственным постоянным идентификатором.

Для первой реализации принят формат:

```text
<category>.<name>

robot.guardian
module-core.chain-lightning
technology.salvage-protocol
```

- Используются только строчные ASCII-буквы, цифры, дефис и одна точка между категорией и именем.
- ID не зависит от имени файла, отображаемого текста или локализации.
- После попадания контента в сохранения ID не переименовывается; замена оформляется миграцией save data.
- Уникальность проверяется в Editor validation tooling до сборки контента.
- Runtime-состояние и сохранения ссылаются на definition по ID, а не сериализуют Unity asset reference как единственный ключ.

## Тестирование

- Edit Mode: доменная логика, расчёты, правила переходов, сериализация и валидация данных.
- Play Mode: интеграция MonoBehaviour, prefab, сцен, UI и Unity lifecycle.
- Перед завершением задачи: компиляция завершена, Console проверена, релевантные тесты выполнены.

## Открытые решения

- 2D или 3D как окончательное направление.
- Дополнительные целевые платформы после PC и момент начала их поддержки.
- Ручное, процедурное или смешанное построение уровней.
- Структура метапрогрессии и сохранений.
- Точный способ composition root и внедрения зависимостей.
- Нужны ли Addressables и асинхронная доставка контента.
